# Hallucination

When a model generates responses that aren't grounded in facts. Along with [[Inconsistency]], it's the fundamental challenge of building reliable AI applications. One-fifth of customer support requests at one AI company were about handling inconsistency.

## Two Hypotheses

### 1. Self-Delusion (DeepMind, Ortega et al., 2021)

The model can't distinguish between data it was *given* and data it *generated*. Example:

> Prompt: "Who's Chip Huyen?"
> Model generates: "Chip Huyen is an architect."
> Now the model treats "Chip Huyen is an architect" as a given fact and builds on it.

This leads to **snowballing hallucinations** (Zhang et al., 2023) — an initial wrong assumption causes the model to make more errors, even on questions it could otherwise answer correctly.

**Mitigation**: RL techniques that make the model differentiate between user prompts (observations) and its own outputs (actions). Also, including factual/counterfactual signals in training data.

### 2. Knowledge Mismatch (Leo Gao, OpenAI; John Schulman, OpenAI)

During SFT, models are trained to mimic labeler responses. If labelers use knowledge the model doesn't have, we're *teaching the model to hallucinate*.

**Mitigation**: Verification — ask the model to cite sources for each claim. Better reward functions that specifically punish fabrication. Schulman believes LLMs "know if they know something."

## Practical Mitigations

- Prompting: "Answer truthfully. If unsure, say 'I don't know.'"
- Ask for concise responses — fewer tokens = less chance to fabricate.
- [[RAG]] — ground responses in retrieved facts.
- [[Test Time Compute]] — generate multiple responses, pick the most consistent.
- [[Preference Finetuning]] — mixed results (RLHF improved overall quality but *worsened* hallucination for InstructGPT).

## Historical Note

Hallucination in text generation was documented as early as 2016 (Goyal et al.), predating the transformer architecture and the term "foundation model."

## Evaluation (Ch.3–4)

Chapters 3 and 4 significantly expand on how to detect and measure hallucination through [[Factual Consistency]] evaluation:

- **Local factual consistency**: Verify outputs against a provided context (crucial for [[RAG]], summarization, customer support)
- **Global factual consistency**: Verify outputs against open knowledge (for general chatbots, fact-checking)
- **SelfCheckGPT**: Generate multiple outputs; if they disagree, the original is likely hallucinated
- **SAFE** ([[DeepMind]]): Decompose output into individual statements → search engine verification
- **Textual entailment**: Classify (premise, hypothesis) pairs as entailment/contradiction/neutral
- **TruthfulQA benchmark**: 817 questions humans commonly answer incorrectly due to false beliefs. Human expert baseline: 94%.

> When designing hallucination metrics, analyze which query types the model is most likely to hallucinate on — niche knowledge and queries about things that don't exist are particularly vulnerable.

## RAG and Agents (Ch.6)

Chapter 6 reinforces RAG as a primary hallucination mitigation and reveals new hallucination vectors in agentic systems:

- **RAG reduces hallucination** by grounding responses in retrieved facts. Lewis et al. (2020) found that relevant information access helps models "generate more detailed responses while reducing hallucinations."
- **Agent hallucination**: In agentic systems, both the action sequence and parameters are generated by AI and can be hallucinated — calling invalid functions or valid functions with wrong parameters. This is a unique failure mode beyond text-based hallucination.
- **Compound hallucination risk**: Agent accuracy degrades multiplicatively across steps (95% per step → 0.6% over 100 steps), amplifying hallucination risks in multi-step tasks.
- **Web browsing trade-off**: Gives agents up-to-date information (reducing staleness hallucination) but also exposes them to internet misinformation.

## Output Guardrails (Ch.10)

Chapter 10 treats hallucination as a production architecture problem — something caught and handled by the system, not just a model property:

- **Factual consistency guardrails**: output guardrails check model responses for consistency with retrieved context or known facts before delivery to the user. Violations trigger retry logic or human fallback.
- **AI-as-a-judge integration**: [[AI as a Judge]] evaluators run on model outputs in production, scoring factual consistency in real time.
- **Observability**: factual consistency score is tracked as a key system metric alongside format failure rate and toxicity rate.
- **Retry logic**: when a hallucination is detected, the system can re-prompt the model (often with an explicit instruction to avoid the failing pattern) before returning an error.

## Finetuning and Hallucination (Ch.7)

- **SFT can induce hallucination**: if supervised finetuning examples contain claims the model can't verify, the model learns to produce similar confident-but-wrong statements.
- **Prompt loss masking**: by masking the loss on instruction tokens, the model is not penalized for not "knowing" the instruction — which reduces one mode of hallucination induction.
- **Synthetic data risk**: model distillation and AI-powered data synthesis can perpetuate teacher model hallucinations at scale. See [[Data Synthesis]] for model collapse risks.
